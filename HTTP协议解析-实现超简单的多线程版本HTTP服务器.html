HTTP协议解析-实现超简单的多线程版本HTTP服务器

发表于 2018-08-03 | 分类于 Linux ， network

HTTP协议是一个应用层的协议。本文讲述HTTP你协议的格式、方法属性以及实现一个简单的HTTP服务器，该服务器在网页上显示一句Hello World。以下总结了一些HTTP的特点：

  • 通常采用客户服务器端的模式
  • HTTP服务器的程序规模小，通信速度很快
  • HTTP可以通过Content-Type字段指定文本类型，所以灵活
  • 无连接，每次连接只处理一个请求，服务器处理完客户的请求，并且收到了客户端的应答之后，断开连接。采用这种方式可以节省传输时间。
  • 无状态，HTTP协议每当有新的请求产生，就会产生新的响应。协议本身不保留之前的请求或响应，这样是为了更快的处理大量的事务，保证协议的可伸缩性。但是随着网络的发展，业务逻辑变得复杂，无状态处理起来便的麻烦，在1.1版本的HTTP协议中，虽然还是无状态的，但是采用了cookie技术，用来保存用户的部分信息。

URL

URI和URL

在认识HTTP之前，我们有必要认识URL。我们常说的网址就是一类URL。URL的全称为统一资源定位符。其实还有一个叫做URI的名词，URI指的是统一资源标识符。

两者有什么区别呢？其实URL包含了URL。URI指的是网络上的一种资源，通过它不一定能够找该资源。URL指的是具体的资源，通过它一定可以找到该资源。URL是URI的子集。举个例子，URI就像是身份证上的名字张三，通过张三这个名字我们不一定能找到张三；URL就是身份证上的地址，我们通过这个地址可以找到张三这个人。

URL的格式

  • 使用HTTP协议我们的URL的协议方案名就需要是http，其次登陆信息通常是省略的。
  • 接下来就是域名或者IP地址，之后用:指定端口号，这个端口号跟使用的协议有关。当前使用的http协议对应端口号就是80，https使用的端口号就是443。
  • 在这之后通过/表示分隔，这里一直到?之前表示的是访问资源的绝对路径。如果不添加一般默认访问首页。
  • ?之后表示的传入的参数，每个参数用键值对的形式表示，同时用&进行分隔。

HTTP请求格式

上图中有一个简单的请求格式，就是我输入的GET / HTTP/1.0，现在具体说一下浏览器通过域名或IP访问资源的时候的HTTP格式。

  • 第一行称为请求行，其中按照顺序包含了：请求方法：通常是GET或POST方法；请求URL：也就是我们在地址栏中输入的信息；HTTP版本信息。这三个元素通过空格分隔开来。
  • 第二行一直到空行为止，称为请求报文：这里有若干行，他们以行为单位。每一行的格式都是属性名： 属性类型，在冒号之后有一个空格。他们的作用是指定了报文的属性，请求连接的属性。
  • 接下来是空行：空行是必须存在的，他用来分隔请求正文和报文。就算请求没有正文空行也是必须存在。
  • 最后一个部分是报文正文：通常我们使用GET请求方式是没有正文的。一般来说POST请求的时候会在报文正文中写入传入的参数。

HTTP响应格式

  • 第一行是响应行，其中按照顺序包括了：HTTP版本：指定了使用的版本信息；响应状态码：是一个三位数字，成功是200，通常还有3xx、4xx、5xx；状态描述语：这个跟状态码是一起的，用语言进行描述给用户看。这三个用空格分开。
  • 第二行一直到空行为止，称为响应报文。跟请求报文一致
  • 必有的空行，用来分隔。
  • 最后部分响应正文：这里带回的就是我们请求的资源，通常是HTML格式，用于浏览器解析展示或者是一个下载资源等。

HTTP常见的请求方法

我们进行HTTP请求的时候，需要指定方法，常用的有GET、POST、HEAD等。

  • GET：获取资源方法；该方法用来请求访问已被URI十倍的资源，指定的资源经过服务器端解析之后返回响应的内容。通过通过GET方法传入参数的话是在URL中通过?进行开始传入，形式是键值对，多个参数之间通过&连接。
  • POST：传输实体主体。POST的方式和GET方式都可以获得资源，但是POST的安全性更高。因为传入的参数不在URL中显示写入，而是通过报文正文传输。
  • PUT：传输文件；客户端给服务器端传输文件，然后保存在URL指定的位置。但是这个存在安全隐患，所以通常被服务器端禁止使用。
  • HEAD：请求头部；通过这个方法，只会获取到报文的头部信息，不会返回正文。通常用这个方法来确认URL的有效性以及资源更新的日期。
  • DELETE：删除文件；删除服务器端保存的资源，禁用~
  • OPTIONS：查询；通过这个方法知道该服务器端支持的方法是什么。
  • CONNECT：要求使用隧道协议连接代理；CONNECT 方法要求在与代理服务器通信时建立隧道，实现用隧道协议进行 TCP 通信。主要使用 SSL（Secure Sockets Layer，安全套接层）和 TLS（Transport Layer Security，传输层安全）协议把通信内容加密后经网络隧道传输。

HTTP常见的头部信息

回忆我们HTTP协议格式，从第二行开始一直到空行，有若干行的键值对，他们以行为单位。他们就是指定报文属性的头部信息。常见的头部信息如下：

  • Content-Type：指定数据类型(text/html)；通过这个参数就可以知道传输或者返回的数据是什么类型的。
  • Content-Length：报文正文长度；这个参数指定了报文正文的长度，它让我们知道了当前的HTTP最终的结束问题。
  • Host：客户端告知服务器，请求的资源在服务器哪个主机的哪个端口上。
  • User-Agent：声明用户的操作系统和浏览器版本信息。
  • Referer：当前页面是从哪个页面跳转过来的。
  • Location：搭配状态码3xx表示客户端接下来将要去哪里访问资源，这是重定向功能。
  • Cookie：用来在客户端存储少量的信息，通常用于实现会话的功能。

HTTP响应的状态码及其描述

我们进行了请求之后，服务器给我们进行HTTP响应，响应的时候，需要返回一个状态码及其描述。状态码可以知道服务器端是否有正确处理请求，如果出错了那是什么原因导致的。常见的状态码如下：

  • 1xx ；接收的请求正在处理
  • 2xx；表示服务器端正确处理了请求
  • 200 OK；正常处理了客户端的请求
  • 204 No Content；请求被正确处理了，当时响应信息中没有响应正文
  • 206 Rartial Content；客户端进行了范围请求，服务器成功执行了这个范围请求。通常用于断点续传
  • 3xx；表示重定向
  • 301 Moved Permanently；永久性重定向，HTTP首部的Location字段被更新为新的URL
  • 302 Found；临时性重定向，本次使用重定向的新地址。下次用回原来的。301和302就算使用POST方法，通常浏览器也会改为GET方法，虽然这个不符合标准。
  • 307 Temporary Redirect；临时重定向。307会遵守标准不会冲POST变成GET。
  • 4xx；客户端错误
  • 400 Bad Request；请求报文中存在语法错误。
  • 403 Forbidden；访问了被禁止访问的资源
  • 404 Not Found；访问的资源不存在于服务器上
  • 5xx；服务器错误
  • 500 Internal Server Error；服务器端在执行请求的时候出现了错误。
  • 503 Service Unavailable；服务器暂时超负荷或者正在进行停机维护

HTTP服务器

将服务器代码编译运行之后，执行服务器。然后通过跟服务器连入了同一个WiFi的手机或电脑进行访问。访问的时候需要加入端口号，如：http://192.168.1.104:8888。其中192.168.1.104是服务器的IP地址。

该HTTP服务器接收了客户端的请求，给予回应并受到确认之后就将连接断开，这个是正常的处理方式。但是现在观察我们的HTTP首部字段，里面有一个属性：Connection:keep-alive。该属性表示保持连接。当客户端有多个请求的时候，就不需要频繁的创建连接了。
